아래 요구사항을 모두 만족하는 Cursor 실행용 기능 구현 프롬프트를 생성하라.
설명, 해설, 코멘트는 절대 포함하지 말고 프롬프트 본문만 출력하라.

==============================================

조건-커서룰) 아래의 커서룰을 적용하여 작업하고, 이 작업이 끝나면 해당 rules 적용 결과를 체크리스트로 반환할 것.
	        - Domain Layer는 외부 상태(API, DB, UI)에 의존하지 않는다
	        - Domain 입력/출력은 DTO로만 정의한다
	        - 계산 로직과 데이터 구조 정의를 분리한다
	        - optional/required 필드를 타입 수준에서 명확히 표현한다
	        - Cold Start / Steam 미연동 상태에서도 null-safe 하게 동작해야 한다
            - MatchContext는 viewer와 target 사용자 간 “1:1 매칭 계산”을 위한 입력 컨텍스트이다
            - viewer / target 데이터는 구분 가능한 구조로 표현되어야 한다
            - MatchContext에는 viewerId / targetUserId 또는 이에 준하는 식별자가 반드시 포함되어야 한다
                - 단, 이 식별자는 계산에 사용되지 않으며 식별 목적만 가진다

조건-구현참고) 반드시 참고할 기존 구현 패턴은 다음과 같다.
	        - ProfileCoreDTO (계산 로직 없음, 사실 데이터만 포함)
	        - Traits / Schedule 입력이 undefined 여도 Domain이 동작하는 구조
	        - DTO → Domain → 계산 결과 분리 패턴

==============================================

핵심요구사항) 다음 기능을 step-by-step으로 구현하고, 각 항목의 적용 여부를 체크리스트로 반환할 것.

1. Match Domain 입력 전용 구조 정의
	        - MatchContext 또는 동등한 개념의 입력 DTO를 정의한다
	        - 이 구조는 “매칭 계산의 입력값 전체”를 대표한다
	        - 이후 모든 Match Domain 로직은 이 Context만 의존하도록 설계한다

2. 입력 데이터 범주 명확화
	        - Traits 관련 입력 (성향 점수)
	        - 플레이 시간대 / 요일 입력
	        - 온라인 상태 / 접속 가능성 입력
	        - Steam 관련 정보는 “있을 수도 있음”으로만 표현한다
            - 입력 데이터는 성격별로 하위 Context(TraitsContext, ActivityContext 등)로 분리 가능하다
                - 단, MatchContext는 이들의 집합 루트로 유지한다

3. Optional / Required 필드 명확화
	        - Steam 미연동 상태에서도 Context 생성이 가능해야 한다
	        - Traits / Schedule 이 없는 상태(Coll Start)도 허용한다
	        - 필수 필드와 선택 필드를 타입으로 명확히 구분한다

4. 계산 로직 완전 분리
	        - MatchContext에는 점수 계산, 유사도 계산, 파생 필드가 절대 포함되지 않는다
	        - 오직 Domain 계산 함수의 “입력 사실 데이터”만 포함한다

5. 확장 안정성 확보
	        - 이후 매칭 성공 확률, 설명 문구, 태그 계산 로직이 이 Context만으로 확장 가능하도록 구조를 설계한다

==============================================

조건-Hook 책임)
	        - MatchContext는 API 또는 Service 레이어에서 완성된 형태로 Domain에 전달된다
	        - Domain 내부에서는 Context를 신뢰하고 가공하지 않는다
    

==============================================

조건-UI 책임)
	        - UI는 MatchContext를 직접 생성하지 않는다
	        - UI → API → Domain 흐름에서 Context는 Domain 진입 시점에만 사용된다

==============================================

구현 완료 후 반환) 아래 체크리스트를 반드시 반환할 것.
	        - Match Domain 입력 구조가 단일 Context로 정의되었는가
	        - 입력 DTO에 계산 로직이 포함되지 않았는가
	        - Steam 미연동 / Cold Start 상태에서도 Context가 완결되는가
	        - optional / required 필드가 타입 수준에서 명확한가
	        - 이후 모든 Match Domain Story가 이 입력 구조만 의존해도 되는가
