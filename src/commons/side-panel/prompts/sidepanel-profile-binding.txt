	        - 새 의존성 추가 금지.
	        - 구현 후 lint/typecheck에 걸리지 않게 정리한다(불필요 import 제거).

핵심요구사항) 다음 기능을 step-by-step으로 구현하고, 각 항목의 적용 여부를 체크리스트로 반환할 것.

{{CORE_REQUIREMENTS}}
	1.	userId 전달 경로 확정 및 연결

	•	Side Panel이 열릴 때 “대상 유저 userId”를 확실히 받을 수 있도록 한다.
	•	userId는 다음 중 하나로 전달한다(코드베이스에 맞게 최적 선택):
	•	props로 전달(권장: SidePanel open 트리거에서 userId를 함께 전달)
	•	route param으로 전달(이미 SidePanel이 라우팅 기반이면 이 방식)
	•	UI에서 “내 userId”를 만들거나 추출하는 로직 추가 금지. (auth.uid, supabase auth 호출 금지)

	2.	/api/profile/:userId fetch 로직 구현 (Hook 또는 Container 레벨)

	•	새 훅(또는 컨테이너) 생성: 예) useProfileByUserId(userId: string | undefined)
	•	내부에서 endpoint를 정확히 호출: GET /api/profile/${userId}
	•	userId가 undefined/빈 문자열이면 요청하지 않고 상태를 empty로 둔다(불필요 요청 방지).
	•	상태 모델을 명확히 정의(예시):
	•	loading: boolean
	•	viewModel?: ProfileViewModel
	•	error?: { status: 403 | 404 | 500 | 401 | number }
	•	empty: boolean (또는 viewModel undefined로 대체)
	•	응답 처리 규칙:
	•	200: JSON을 ProfileCoreDTO로 받고, toProfileViewModel(dto) 호출하여 viewModel 세팅
	•	403: 권한 없음 상태로만 표시(문구 단순 처리)
	•	404: 없음 상태로만 표시(문구 단순 처리)
	•	그 외(500 포함): 일반 에러 상태로만 표시(문구 단순 처리)
	•	“API 에러 문구 가공” 금지:
	•	response.json()의 message를 UI 친화적으로 변환하지 말 것
	•	UI에는 단지 상태(Forbidden/NotFound/Error)만 전달
	•	domain 변환 함수(예: toRadarData 등)를 UI에서 직접 호출 금지.
	•	ViewModel 변환은 오직 toProfileViewModel 단일 함수로 수행.

	3.	Side Panel UI 분기 구현 (권한/404/기타)

	•	Side Panel 컴포넌트는 훅/컨테이너에서 내려주는 상태를 기반으로만 렌더링 분기한다.
	•	분기 최소 요구:
	•	loading: 스피너/스켈레톤/텍스트 “Loading” 등
	•	403: “Forbidden” 또는 “Access denied” 수준의 단순 문구
	•	404: “Not found” 수준의 단순 문구
	•	기타 에러: “Error” 수준의 단순 문구
	•	성공: ProfileViewModel 기반 UI 렌더
	•	UI는 endpoint/path, status code, auth 관련 세부 구현을 알 필요가 없도록 구성한다.

	4.	기존 mock/하드코딩 데이터 제거(해당 시)

	•	Side Panel에 프로필 mock 데이터/상수/임시 state가 있으면 제거하고, 훅 결과로 대체한다.
	•	단, ViewModel 구조 변경은 금지. (기존 UI가 요구하는 ViewModel 필드 유지)

	5.	코드 품질/안전장치

	•	fetch 중복 호출 방지(의존성 배열/abort controller 등, 기존 패턴과 일치)
	•	컴포넌트 unmount 시 setState 경고 방지(기존 패턴 준수)
	•	타입 단에서 ProfileCoreDTO/VM 경계 유지:
	•	UI 컴포넌트 props로 ProfileCoreDTO를 넘기지 말고 ProfileViewModel만 넘긴다.

조건-Hook 책임)
{{HOOK_SCOPE}}
	•	Hook/Container가 담당:
	•	endpoint 호출: /api/profile/:userId
	•	status 기반 상태 매핑(loading/forbidden/notFound/error/success)
	•	성공 시 toProfileViewModel(dto) 단일 호출로 ViewModel 생성
	•	Hook/Container가 담당하지 말아야 할 것:
	•	auth 로직(내 userId 계산, 토큰 발급, 로그인 체크)
	•	UI 문구 가공(서버 message를 재작성/해석)
	•	도메인 변환 함수(toRadarData 등) 직접 호출(반드시 toProfileViewModel만)

==============================================

조건-UI 책임)
{{UI_SCOPE}}
	•	UI 컴포넌트가 담당:
	•	userId를 props/route param으로 “전달받아” 훅에 넘김
	•	훅의 상태에 따라 렌더링 분기
	•	성공 시 ViewModel로만 화면 구성
	•	UI 컴포넌트가 하면 안 되는 것:
	•	/api/profile/:userId 외 endpoint 사용
	•	auth 로직 추가
	•	API 응답 가공/해석(문구 생성/상태 판정 고도화)
