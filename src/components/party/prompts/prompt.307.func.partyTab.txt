핵심요구사항) 다음의 기능을 playwright 테스트를 활용하여 TDD기반으로 구현하고, 테스트에 통과할 때까지 반복할 것.
            1) 테스트 제외 라이브러리
                - jest
                - @testing-library/react

            2) 테스트 조건
                - timeout은 설정하지 않거나, 500ms 미만으로 설정할 것.
                - 모든 페이지의 로드 식별 조건은 아래를 따를 것.
                    - 페이지 로드 식별 요구사항: 고정식별자 data-testid 대기 방법
                    - 페이지 로드 식별 금지사항: networkidle 대기 방법

            3) 테스트 데이터 조건
                - 실제데이터를 사용할 것.
                - Mock데이터 사용하지 말 것.
                - 데이터베이스의 실제 party_posts, party_members 테이블 데이터를 사용할 것.

            4) 테스트 데이터타입
                - 데이터베이스 테이블: party_posts, party_members
                - party_posts 테이블 필드:
                    - id: number
                    - game_title: string
                    - party_title: string
                    - description: string
                    - creator_id: string (파티 생성자 user_id)
                    - 기타 필드들...
                - party_members 테이블 필드:
                    - id: number
                    - post_id: number (party_posts.id와 외래키 관계)
                    - user_id: string (참여한 유저의 user_id)
                    - role: string (역할, 예: 'member')
                    - joined_at: string (참여 시각)
                    - 주의: party_members에 해당 파티에 참여한 멤버 정보가 저장됨
                - 인증 정보:
                    - 로그인한 유저의 user_id는 useAuth 훅을 통해 가져올 수 있음
                    - useAuth는 @/commons/providers/auth/auth.provider에서 제공됨
                    - user.id를 통해 현재 로그인한 유저의 user_id를 확인할 수 있음

            5) 테스트시나리오(모든유저)
                1. /party에 접속하여 페이지 로드 확인
                2. 하단 유저시나리오에 적절한 테스트시나리오를 생성하여 진행

핵심요구사항) 다음의 조건에 맞게 파티 탭 기능을 step-by-step 으로 구현하고, 적용 결과를 체크리스트로 반환할 것.
            1) 유저시나리오(전체 파티 탭)
                1. /party 페이지 접속 시 기본적으로 "전체 파티" 탭이 선택되어 있는지 확인
                2. "전체 파티" 탭이 선택된 상태에서 모든 파티 게시물이 표시되는지 확인
                    - 필터링 로직:
                        1) activeTab이 'all'인 경우: 모든 party_posts 데이터를 표시
                        2) 기존 필터(장르, 검색)와 함께 동작해야 함
                        3) party_members 테이블과의 조인 없이 모든 파티를 표시
                    - 기본값: activeTab의 초기값은 'all'로 설정

            2) 유저시나리오(참여 중인 파티 탭)
                1. "참여 중인 파티" 탭 버튼 클릭
                2. 탭이 전환되고, 로그인한 유저가 참여한 파티만 표시되는지 확인
                    - 필터링 로직:
                        1) activeTab이 'participating'인 경우: party_members 테이블에서 현재 로그인한 유저의 user_id와 일치하는 레코드 조회
                        2) party_members.user_id === 현재 로그인한 유저의 user_id인 경우의 post_id를 찾음
                        3) 해당 post_id와 일치하는 party_posts만 표시
                        4) 로그인하지 않은 경우: 빈 목록 표시 또는 로그인 유도 메시지 표시
                    - 인증 확인:
                        1) useAuth 훅을 사용하여 현재 로그인한 유저 정보 확인
                        2) user.id를 통해 user_id 가져오기
                        3) user가 null인 경우 처리 필요
                    - 기존 필터(장르, 검색)와 함께 동작해야 함 (AND 조건)
                    - 예시: 
                       - 로그인한 유저의 user_id가 'user-123'인 경우
                       - party_members에서 user_id='user-123'인 레코드의 post_id를 조회
                       - 해당 post_id에 해당하는 party_posts만 표시

            3) 유저시나리오(탭 전환)
                1. "전체 파티" 탭에서 "참여 중인 파티" 탭으로 전환
                2. "참여 중인 파티" 탭에서 "전체 파티" 탭으로 전환
                3. 각 탭 전환 시 올바른 파티 목록이 표시되는지 확인
                    - 탭 전환 시:
                        1) activeTab 상태가 올바르게 업데이트되는지 확인
                        2) 탭에 따른 필터링이 즉시 적용되는지 확인
                        3) 무한 스크롤이 올바르게 동작하는지 확인
                    - 탭 전환 시 기존 필터(장르, 검색) 상태는 유지되어야 함
                    - 예시:
                       - "전체 파티" 탭에서 "액션" 장르 필터 적용 후 "참여 중인 파티" 탭으로 전환
                       - "참여 중인 파티" 중에서도 "액션" 장르에 해당하는 파티만 표시되어야 함
