조건-커서룰) 아래의 커서룰을 적용하여 작업하고, 이 작업이 끝나면 해당 rules 적용 결과를 체크리스트로 반환할 것.
	        - 단일 Service 함수로 구현할 것
	        - Service는 데이터 조회 + 최소 조립 책임만 가진다
	        - Repository는 이미 구현되어 있다고 가정하고 재구현하지 않는다
	        - 예외는 throw 로 처리하되, 메시지 포맷/UI 기준은 고려하지 않는다
	        - 타입 안정성을 최우선으로 한다 (ProfileCoreDTO 기준)
	        - 불필요한 로직, boolean flag, UI 전용 분기 로직 추가 금지

조건-구현참고) 반드시 참고할 기존 구현 패턴은 다음과 같다.
	        - Service 레벨에서 여러 Repository 결과를 조합하는 패턴
	        - Repository는 findByUserId(userId: string) 형태의 단일 조회 함수
	        - DTO는 Service에서만 조립하며, Repository는 DB row 그대로 반환

==============================================

핵심요구사항) 다음 기능을 step-by-step으로 구현하고, 각 항목의 적용 여부를 체크리스트로 반환할 것.
	1.	입력으로 userId: string을 받는 getMyProfileService 함수 구현
	2.	userProfilesRepository.findByUserId(userId) 호출
	3.	profile row가 없을 경우 즉시 error throw
	4.	userTraitsRepository.findByUserId(userId) 호출
	5.	userPlaySchedulesRepository.findByUserId(userId) 호출
	6.	traits와 schedule 모두 없는 경우
	•	정상 흐름
	•	ProfileCoreDTO에 traits, schedule을 undefined로 설정
	7.	traits만 있거나 schedule만 있는 경우
	•	데이터 불일치로 판단
	•	error throw
	8.	traits와 schedule이 모두 있는 경우
	•	ProfileCoreDTO에 그대로 매핑
	9.	nickname, animalType은 DB 값이 없을 경우 undefined 허용
	10.	Service는 크래시 없이 명시적 throw만 수행
	11.	반환 타입은 ProfileCoreDTO로 고정
	12.	DB row → DTO 조립까지만 수행
	13.	UI/ViewModel/매칭/판단 로직 일절 포함하지 않음
