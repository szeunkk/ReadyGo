아래 조건을 모두 적용하여, 아래의 요구사항을 모두 구현할 것.
구현 결과를 체크리스트로 반환할 것.

==============================================

조건-커서룰) 아래의 커서룰을 적용하여 작업하고, 이 작업이 끝나면 해당 rules 적용 결과를 체크리스트로 반환할 것.
	        - API 레벨(Route / Controller)에서만 에러 매핑을 수행할 것
	        - Service 레이어에는 try/catch를 추가하지 말 것
	        - Service에서 throw된 Error 타입을 기준으로 HTTP 상태 코드를 결정할 것
	        - 알 수 없는 에러는 반드시 fallback 처리할 것
	        - 에러를 swallow하거나 로그만 찍고 응답을 종료하지 말 것

==============================================

조건-파일경로) 참고 및 기준 파일 경로는 다음과 같다.
	        - app/api/profile/**/route.ts

조건-파일경로) 참고할 profileErros 파일 경로
	        - src/commons/errors/profile/profileErrors.ts

==============================================

조건-구현참고) 반드시 참고할 기존 구현 패턴은 다음과 같다.
	        - 기존 Route Handler에서 Service 호출 후 try/catch로 에러를 잡아 HTTP Response를 반환하는 패턴
	        - Error instanceof 또는 error.name 기반 분기 처리
	        - JSON 응답 구조를 명시적으로 구성하는 방식

==============================================

핵심요구사항) 다음 기능을 step-by-step으로 구현하고, 각 항목의 적용 여부를 체크리스트로 반환할 것.

1. Service 에러 → HTTP 상태 코드 매핑
	        - ProfileNotFoundError 발생 시 HTTP 404 반환
	        - ProfileDataInconsistencyError 발생 시 HTTP 500 반환
	        - ProfileFetchError 발생 시 HTTP 500 반환
	        - 위에 해당하지 않는 모든 에러는 HTTP 500 fallback 처리

2. 에러 응답 포맷 통일
	        - 모든 에러 응답은 JSON 형태로 반환할 것
	        - 응답 body는 반드시 다음 필드를 포함할 것
	        - code: string
	        - message: string
	        - code 값은 Error 클래스명 또는 사전에 정의된 에러 코드 문자열을 사용할 것
	        - message는 Service 에러의 message를 그대로 전달할 것

3. API 레벨 책임 고정
	        - Route Handler에서만 에러 매핑 로직을 구현할 것
	        - Service 내부 로직, throw 지점, 에러 타입 정의는 수정하지 말 것

==============================================

조건-Hook 책임)
	        - 해당 없음 (Hook 로직 구현 ❌)

==============================================

조건-UI 책임)
	        - UI 친화적 메시지 생성 ❌
	        - 다국어 처리 ❌
	        - UX 기준 분기 ❌

==============================================

구현 완료 후 반환) 아래 체크리스트를 반드시 반환할 것.
	        - ProfileNotFoundError → 404 매핑됨
	        - ProfileDataInconsistencyError → 500 또는 409 매핑됨
	        - ProfileFetchError → 500 매핑됨
	        - Unknown Error의 code는 INTERNAL_ERROR를 사용
	        - Service 레이어에 try/catch 추가되지 않음
	        - 에러 swallow 없음
	        - 모든 에러 응답이 { code, message } 포맷을 따름
	        - API 레벨에서만 에러 처리 책임을 가짐
