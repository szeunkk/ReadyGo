	•	DTO는 UI 컴포넌트 폴더 내부에 존재해서는 안 된다.
	•	API / Service / FE 어디에서도 동일 타입을 import 하여 사용 가능해야 한다.
	•	런타임 로직, 계산, 변환 함수는 포함하지 않는다.
	•	타입 변경 시 영향 범위를 명시적으로 주석으로 남긴다.

조건-구현참고) 반드시 참고할 기존 구현 패턴은 다음과 같다.
	•	commons/types 하위 DTO는 interface 또는 type alias 만 정의한다.
	•	DB 컬럼/도메인 타입과 1:1 매핑 가능한 구조를 유지한다.
	•	optional + null 가능성은 타입 레벨에서 명시한다 (?: T | null)

==============================================

핵심요구사항) 다음 기능을 step-by-step으로 구현하고, 각 항목의 적용 여부를 체크리스트로 반환할 것.
	1.	ProfileCoreDTO 타입을 새로 정의한다. 이 때, 필요한 테이블과 컬럼은 아래를 참고할 것.
        •	user_profiles → nickname, animal_type
        •	user_traits → 5개 trait score
        •	user_play_schedules → day_type[], time_slot[]
        필수 필드 vs 선택 필드 구분
        •	필수: userId (uuid), nickname(없으면 ‘’/fallback)
        •	선택: schedule(없으면 빈 배열), animalType(없으면 빈 값/unknown 정책), traits(5개 모두)
}
	2.	Steam, 외부 플랫폼, 인증 관련 필드는 절대 포함하지 않는다.
	3.	내 프로필 / 상대 프로필 모두에서 재사용 가능한 구조인지 검증한다.
	4.	필수 필드와 선택 필드를 명확히 구분한다.
	5.	DB 값이 없는 경우의 정책을 타입/주석으로 명확히 표현한다.
	6.	DTO 변경 시 영향을 받는 레이어(API, Service, FE)를 주석으로 명시한다.

==============================================
